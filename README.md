# CS-305
CS 305 Project 2
What did you do well when you found your client’s software security vulnerabilities? Why is it important to code securely? What value does software security add to a
company’s overall well-being?
I was thorough in identifying vulnerabilities across multiple layers of the application — from the missing HTTPS configuration and lack of any data integrity verification, to reviewing third-party dependencies for known CVEs using the OWASP Dependency-Check tool. I did well at prioritizing which vulnerabilities were introduced by my own refactoring versus which were pre-existing in the base dependencies, which kept the scope of remediation focused and accurate. Coding securely is critical because vulnerabilities in financial software can lead to data breaches that expose sensitive client information, result in regulatory penalties, and permanently damage a company’s reputation. Software security adds direct value to a company’s well-being by reducing breach risk, supporting compliance with regulations like GLBA and PCI-DSS, and building the client trust that financial advisory businesses depend on entirely.
Which part of the vulnerability assessment was challenging or helpful to you?
The most helpful part of the assessment was using the OWASP Dependency-Check tool to scan project dependencies against the NIST National Vulnerability Database. It made visible the CVEs that existed in the base Spring Boot dependencies and gave me a clear, evidencebased way to distinguish pre-existing vulnerabilities from anything I might introduce through refactoring. The most challenging part was ensuring the SSL/TLS configuration was complete end-to-end — generating the keystore with the Java Keytool, placing it correctly in the classpath, and verifying that all four application.properties fields (key-alias, key-storepassword, key-store, key-store-type) were correctly filled in so the application would actually start on HTTPS rather than failing silently.
How did you increase layers of security? In the future, what would you use to assess vulnerabilities and decide which mitigation techniques to use?
I increased security in multiple complementary layers following a defense-in-depth approach. At the transport layer, I configured TLS/SSL by generating a PKCS12 keystore and converting the application from HTTP to HTTPS on port 8443, encrypting all data in transit. At the data integrity layer, I implemented SHA-256 hashing via Java’s MessageDigest API to provide checksum verification, allowing any party to confirm that transmitted data has not been altered. At the code quality layer, I used only standard Java SE cryptographic libraries, implemented thread-safe per-request MessageDigest instantiation, and added explicit exception handling. In the future I would continue using the OWASP Dependency-Check tool for static dependency analysis, combined with manual code review guided by the OWASP Top 10 and the Vulnerability Assessment Process Flow to systematically evaluate input validation, cryptography, client/server security, error handling, code quality, encapsulation, and API security. For deciding mitigation techniques, I would weigh CVSS severity scores, exploitability in the specific deployment context, and the effort required to remediate against the risk of leaving a vulnerability unaddressed.
How did you make certain the code and software application were functional and secure? After refactoring the code, how did you check to see whether you introduced new vulnerabilities?
I verified functionality by compiling the refactored code with Maven, running the Spring Boot application, and confirming that the /hash endpoint returned the correct SHA-256
checksum output over HTTPS at https://localhost:8443/hash. I also performed a manual code review to check for syntactical issues, logical errors such as non-thread-safe MessageDigest usage, and security gaps such as weak algorithm selection or user-input injection surfaces. To verify I had not introduced new vulnerabilities through refactoring, I ran the OWASP Dependency-Check Maven plugin and reviewed the output specifically for any CVEs tied to dependencies or libraries added as part of my changes. Because the SHA256 implementation used only java.security.MessageDigest — a standard Java SE library class — no new external dependencies were introduced and no new vulnerabilities were identified.
What resources, tools, or coding practices did you use that might be helpful in future assignments or tasks?
The most reusable tools and practices from this project include: the OWASP DependencyCheck Maven plugin for automated CVE scanning of project dependencies; the Java Keytool utility for generating and managing cryptographic keystores and certificates; the Java MessageDigest API for implementing standards-compliant hashing without relying on thirdparty libraries; and Spring Boot’s built-in SSL configuration via application.properties, which makes enabling HTTPS straightforward in any Spring-based project. The Vulnerability Assessment Process Flow Diagram was also a helpful structured framework for ensuring no security domain was overlooked during review. These tools and the habit of running static analysis after every refactoring cycle will be directly applicable in any future software development role.
